Original Artifact

This is the original inventory management application as it existed at the end of CS 360, before any of the enhancement work for this capstone began. At this stage the app was functional but limited: inventory items were displayed only as bare rows showing a name and quantity, with no way to view further detail or edit an item without going back through the basic list controls. The database layer stored user passwords as plain, unencrypted text, had no fields for category or price, included no timestamp tracking, and used an upgrade method that would permanently delete all existing data if the schema ever changed. There was also no way to search or sort the inventory, so the list became harder to manage as more items were added. This version is included so reviewers can see the true starting point of the project and measure the depth of the improvements made across all three enhancement categories.

Enhancement One — Software Design and Engineering

The original inventory screen only displayed an item's name and quantity in a compact row, with no way to view additional details or make changes without navigating through the basic list controls. This enhancement redesigned that experience by introducing a dedicated, full screen detail view that opens whenever a user taps on an inventory item. The new screen presents the item's name, quantity, category, and price in a clean, organized layout, and includes a color coded stock status banner that immediately communicates whether an item is in stock, running low, or completely out of stock. Users can now also edit an item's name and quantity directly from this screen instead of being routed back through the main list, and every input field across the app was updated with stronger validation to block negative quantities and overly long item names before they ever reach the database.

Enhancement Results

  • Added a full screen item detail view accessible by tapping any inventory item
  • Introduced a color coded stock status banner (in stock, low stock, out of stock) for instant visual feedback
  • Enabled in place editing of item name and quantity directly from the detail screen
  • Strengthened input validation to block negative quantities and excessively long item names
  • Resolved a refresh bug where edited values were not reflected on the inventory list until the onResume method was added

Enhancement Two — Algorithms and Data Structures

The original inventory list had no way to search or sort items, which meant that as the inventory grew, finding a specific item required scrolling through the entire list manually. This enhancement addressed that limitation by introducing a real time search bar and a sort toggle button at the top of the inventory screen. As a user types into the search bar, the list filters instantly to show only matching results, and the sort toggle allows switching between alphabetical order and lowest quantity first, which makes it easy to spot items that need restocking. Both features were implemented at the database query level rather than by filtering data in memory, which keeps performance consistent even as the inventory scales up. While building this feature, a case sensitivity issue was discovered where item names with mixed capitalization or spacing were sorting incorrectly, and that was resolved with a case insensitive sort directive.

Enhancement Results

  • Added a real time search bar that filters the inventory list as the user types
  • Added a sort toggle button to switch between alphabetical order and lowest quantity first
  • Implemented filtering and sorting directly in the SQL query rather than in application memory for better scalability
  • Fixed a case sensitivity bug affecting alphabetical sorting by adding a case insensitive sort directive

Enhancement Three — Databases

The original database had several structural and security weaknesses. User passwords were stored as plain, readable text, the inventory table had no fields for category, price, or creation date, and the schema upgrade method was written in a way that would permanently delete all existing data if the database structure ever changed in the future. This enhancement rebuilt the database layer to address each of these issues. Passwords are now hashed using SHA-256 before they are ever stored, the inventory table was expanded with category, price, and timestamp fields, and the upgrade method was rewritten to preserve existing data during schema changes instead of wiping it out. An Inventory Summary screen was also added that uses SQL aggregate functions to surface useful information at a glance, including total inventory count, items low or out of stock, total inventory value, and the most expensive and most stocked items.

Enhancement Results

  • Implemented SHA-256 password hashing so credentials are never stored as plain text
  • Expanded the inventory schema to include category, price, and creation timestamp fields
  • Rewrote the database upgrade method to preserve existing data instead of deleting it during schema changes
  • Built an Inventory Summary screen using SQL aggregate functions (COUNT, SUM, MAX) to report total items, low and out of stock counts, total inventory value, and top items
